# 会话与复盘协议 (Session & Review Protocol)

> 本文件定义会话开始和结束时的强制执行协议。

## 会话开始协议

每次新会话开始时，必须按以下优先级**自动阅读**文件以恢复状态：

1. [state/context.md](../state/context.md) (核心状态锚点 ⭐)
2. [README.md](../README.md) (项目概览)

## 复盘触发条件

当以下任一情况发生时，**必须**执行复盘流程：

1. 用户触发"复盘"指令或 `/evolve` 命令
2. 任务结束进入总结阶段
3. 记忆系统有变化
4. 每日首次对话

## 复盘流程（五步协议）

### 流程图

```
┌─────────────────────────────────────────────────────────┐
│ 步骤1 [MUST FIRST]: 经验提取                             │
│ → 调用 skill-evolution-manager 或类似工具               │
│ → 输出: 结构化经验总结（JSON）或直接解决问题             │
│ → 验收标准: 问题是否全部解决                             │
│ → 过程中产生有价值的复盘讨论                             │
└─────────────────────────────────────────────────────────┘
                          ↓
⚠️ 禁止跳过！这是复盘的核心价值
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 步骤2 [AFTER STEP 1]: 对话备份                          │
│ → 保存完整会话到 docs/session-log-YYYY-MM-DD.md         │
│ → 包含: 任务阶段 + 复盘讨论                             │
│ → 复盘讨论本身就是高价值内容                            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 步骤3 [DEPENDS ON 1]: 知识持久化                         │
│ → 基于步骤1的输出更新三重文档                           │
│ → work-log.md（过程记录）                               │
│ → context.md（状态锚点）                                │
│ → dialogue-record.md（决策记录）                        │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 步骤4: 记忆系统更新                                      │
│ → 根据步骤1的观察更新记忆文件                           │
│ → MEMORY.md、anti_patterns.md 等                        │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ 步骤5: Git 归档                                         │
│ → Git Commit + Push                                     │
└─────────────────────────────────────────────────────────┘
```

### 执行检查清单

- [ ] 步骤1: 经验提取已完成（问题已解决）
- [ ] 步骤2: 对话已备份（包含复盘讨论）
- [ ] 步骤3: 三重文档已更新
- [ ] 步骤4: 记忆系统已更新（如需要）
- [ ] 步骤5: Git commit + push

### 详细步骤

#### 步骤 1 [MUST FIRST]: 对话备份(原始)

#### 步骤 1 [MUST FIRST]: 经验提取

**目的**：分析问题、提取经验、解决问题

**执行**：
- **触发自主进化**：通过 `Claudeception` (.skills/claudeception/SKILL.md) 规则评估本次会话。
- 调用 `skill-evolution-manager` 或直接分析。
- 提取成功的模式、踩过的坑、流程改进建议。
- 结构化存留至 `.skills/evolution-db.json`（见 Claudeception 要求）。
- **验收标准**：问题是否全部解决且防错经验已被结构化固化。

**场景分类**：

| 场景 | 输出形式 | 验收标准 |
|------|---------|----------|
| Skill 优化 | evolution.json + SKILL.md 更新 | 问题是否解决 |
| 项目级优化 | 直接修改文件 | 问题是否解决 |

**关键理解**：
- 工具（skill-evolution-manager）是手段，不是目的
- 重点是**解决问题**，而不是"必须产出 JSON"
- 过程中产生的复盘讨论本身就是高价值内容

#### 步骤 2 [AFTER STEP 1]: 对话备份

**目的**：保存完整会话（包括复盘讨论）

**执行**：
- 创建 `docs/session-log-YYYY-MM-DD.md`（如不存在）
- 记录完整的用户需求和对话过程
- **包含复盘讨论**：这些讨论本身就是高价值内容
- 保持原始信息，不进行整理或精简

**为什么在步骤1之后**：
- 步骤1（经验提取）会产生有价值的复盘讨论
- 这些讨论也应该被保存到对话备份中
- 一次性保存完整会话，避免重复备份

**模板**：
```markdown
# 对话记录备份

**日期**: YYYY-MM-DD
**项目**: 项目名
**会话目标**: 简述本次对话目标

## 用户需求
> 用户原始需求

## 实施过程
- 探索发现
- 方案设计
- 执行步骤

## 复盘讨论
- 问题分析
- 经验提取
- 方案讨论

## 待办事项
- [ ] 下一步1
- [ ] 下一步2

## 关键配置
| 文件 | 路径 |
```

#### 步骤 3 [DEPENDS ON 1]: 知识持久化

**三重更新**：**基于步骤 1 的提取结果**，更新以下文档：
- `state/work-log.md`（过程与记录）
- `state/context.md`（状态与锚点）
- `state/dialogue-record.md`（关键对话与决策链路）

**技能缝合**（如适用）：通过 `evolution.json` 将经验缝合进对应 Skill 的 `SKILL.md`

**⚠️ 文档格式规范**（必须遵守）：

| 规则 | 要求 | 示例 |
|-----|------|------|
| 时间精度 | 精确到分钟 | 2026-03-01 01:00 |
| 排列顺序 | 倒序（最新在前） | 新内容追加到文件开头 |
| 记录人 | 必须包含 | `*记录人：CC-GLM4.7*` |

**三重文档更新规则**：
- **state/work-log.md**：新内容追加到文件开头（第一个标题后）
- **state/context.md**：新内容追加到"当前状态"之后，"历史状态"开头
- **state/dialogue-record.md**：新内容追加到文件开头（文档说明后）

#### 步骤 4: 记忆系统更新

**目的**：更新项目级记忆文件，确保跨会话的知识连续性

**基于步骤 1 的观察**，更新以下文件（如需要）：
- `.claude/memory/MEMORY.md`（主记忆文件）
- `.claude/memory/project_structure.md`（项目结构）
- `.claude/memory/anti_patterns.md`（反模式手册）
- `.claude/memory/fusion_config.md`（配置融合）

**更新时机**：
- 发现新的反模式
- 项目结构有重大变更
- 配置中心映射有变化
- 工作流程有改进

#### 步骤 5: Git 归档

**Git 提交**：
- 完成所有步骤后，执行标准化 Git Commit 并推送至远程仓库
- 确保环境一致性

## 死锁风险与关键原则

⚠️ 如果跳过步骤1（经验提取），复盘将失去核心价值。

⚠️ 如果跳过步骤2（对话备份），复盘讨论将丢失。

⚠️ **常见错误：把"调用工具"当成"完成复盘"**

| 错误认知 | 正确理解 |
|----------|----------|
| 调用了 skill-evolution-manager = 完成 | **问题是否解决**才是验收标准 |
| 必须产出 JSON 才算复盘 | 项目级优化可以直接改文件 |
| 工具流程僵化执行 | 灵活应用，核心是解决问题 |
| **Hook 通过 = 复盘完成** | **Hook 只是最低门槛，真正验收靠自觉** |

⚠️ **Hook 的局限性**

Hook 只能做静态检查：
- 检查文件是否存在
- 检查内容是否有标志词

Hook **无法验证**：
- AI 是否真的调用了工具
- 问题是否真的被解决
- 是否有深入的思考过程

**务实态度**：
- Hook 通过 ≠ 复盘完成
- Hook 是安全网，不是质量保证
- **真正的验收标准是：你自己的判断**

> **原则**：复盘不仅是总结，更是为了下一次任务的自动化与标准化。
> - **经验提取是核心**（解决问题 > 产出 JSON）
> - 对话备份保留完整会话（包括复盘讨论）
> - 文档更新沉淀知识资产
> - 记忆更新确保跨会话连续性
> - **Hook 检查是最低门槛，真正的质量靠 AI 自觉**

## Git 提交前强制检查

每次 `git commit` 前，必须确认已完成步骤 1-3：

```bash
# Git 提交前检查清单
- [ ] 经验提取已完成，问题已解决（步骤1）
- [ ] 对话已备份（包含复盘讨论）（步骤2）
- [ ] 三重文档已更新（步骤3）
```

**重大开发前额外检查**：
- [ ] 已对比本地文档与 GitHub 远程仓库的差异
- [ ] 确认无冲突或已解决冲突
