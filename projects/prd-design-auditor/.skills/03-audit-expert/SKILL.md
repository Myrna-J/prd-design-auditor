# 03-audit-expert - 审计专家

> **版本**: 1.0.0
> **技能类型**: Core Skill
> **所属项目**: 产品经理AI助理

---

## 🎯 功能定位

对 PRD 文档进行全面审计，识别逻辑漏洞、完整性缺陷、可行性风险，确保需求质量。

---

## 🩺 触发条件 (Symptoms & Triggers)

当以下场景出现时，激活此技能：

- [ ] PRD 文档已完成，准备进入开发阶段
- [ ] 收到供应商提交的 PRD 需要评审
- [ ] 开发过程中频繁出现"需求不明确"问题
- [ ] 需要进行需求变更影响评估
- [ ] 项目回顾时发现需求缺陷需要根因分析

---

## 🛠️ 7步工作流 (7-Step Workflow)

### Step 1: 文档接收与预处理 (Intake)
**目标**: 加载 PRD 文档并提取关键信息

**输入**: `docs/prd/PRD-[功能名]-v[X].md`

**预处理清单**:
- [ ] 确认文档版本号
- [ ] 提取文档元数据（作者、日期、评审人）
- [ ] 识别 PRD 类型（标准/技术/数据/集成）
- [ ] 建立审计上下文（项目背景、业务领域）

**输出**: `audit-context.md`

---

### Step 2: 完整性审计 (Completeness Audit)
**目标**: 检查 PRD 是否包含所有必需章节

**检查清单**（基于 PRD 模板）:

| 章节 | 必须 | 检查项 |
|------|------|--------|
| 文档信息 | ✅ | 版本、日期、作者、评审人 |
| 背景与目标 | ✅ | 问题陈述、成功指标 |
| 用户故事 | ✅ | ≥1 条用户故事，含验收标准 |
| 功能规格 | ✅ | 功能清单、详细规格 |
| 非功能需求 | ⚠️ | 性能、安全、兼容性 |
| 数据模型 | ⚠️ | 涉及数据变更时必须 |
| 用户流程 | ⚠️ | 复杂流程时必须 |
| 界面原型 | ❌ | 有则加分 |
| 发布计划 | ❌ | 大功能时必须 |

**评级**:
- ✅ 完整 - 所有必须项存在
- ⚠️ 部分缺失 - 缺少可选项或部分必须项
- ❌ 严重缺失 - 缺少关键必须项

---

### Step 3: 逻辑一致性审计 (Logic Consistency)
**目标**: 检查 PRD 内部逻辑是否自洽

**审计维度**:

#### 3.1 用户故事一致性
- [ ] 用户角色在全文保持一致
- [ ] 用户故事之间无冲突
- [ ] 验收标准可验证（非主观描述）
- [ ] 优先级标注合理（Must 不过多）

#### 3.2 功能逻辑一致性
- [ ] 输入/输出定义清晰
- [ ] 状态流转完整（无死锁、无遗漏）
- [ ] 边界条件已覆盖
- [ ] 异常处理路径已定义

#### 3.3 数据一致性
- [ ] 数据模型与功能描述一致
- [ ] 字段类型、约束合理
- [ ] 关系定义准确（1:1, 1:N, N:M）

**常见逻辑缺陷**:
```markdown
❌ 矛盾描述
- 前文说"支持批量操作"，后文说"每次只能处理一条"

❌ 循环依赖
- 功能A依赖功能B，功能B又依赖功能A

❌ 不可达状态
- 定义了状态X，但没有任何路径能到达状态X

❌ 遗漏边界
- "用户输入金额"，但未定义最大/最小值
```

---

### Step 4: 可行性审计 (Feasibility Audit)
**目标**: 评估需求的技术和业务可行性

#### 4.1 技术可行性
- [ ] **技术栈匹配**: 现有技术栈能否支持？
- [ ] **性能可行性**: 性能指标是否可达？
- [ ] **集成可行性**: 第三方依赖是否稳定？
- [ ] **安全可行性**: 安全要求是否可实施？

#### 4.2 业务可行性
- [ ] **资源可行性**: 人力/时间是否充足？
- [ ] **合规可行性**: 是否符合法规要求？
- [ ] **运营可行性**: 上线后是否可运维？

**风险评估矩阵**:

| 风险等级 | 定义 | 处理建议 |
|---------|------|---------|
| 🔴 高 | 可能导致项目失败 | 必须解决后才能继续 |
| 🟡 中 | 可能延期或超支 | 需要 mitigation plan |
| 🟢 低 | 影响可控 | 记录并监控 |

---

### Step 5: 规则库审计 (Rule-Based Audit)
**目标**: 使用领域规则库检查业务逻辑

**不动产经营领域规则示例**:

| 规则ID | 规则描述 | 检查方式 |
|--------|---------|---------|
| RE-001 | 租金计算必须考虑免租期 | 搜索"免租期"关键词 |
| RE-002 | 资产盘点必须支持差异调整 | 检查盘点流程是否含调整环节 |
| RE-003 | 合同到期前30天必须预警 | 检查预警机制定义 |
| RE-004 | 费用分摊必须保留计算明细 | 检查分摊逻辑是否可追溯 |

**通用规则**:

| 规则ID | 规则描述 | 严重程度 |
|--------|---------|---------|
| GEN-001 | 所有删除操作必须有二次确认 | 🔴 高 |
| GEN-002 | 敏感操作必须有审计日志 | 🔴 高 |
| GEN-003 | 批量操作必须支持部分失败 | 🟡 中 |
| GEN-004 | 导出功能必须限制数据量 | 🟡 中 |
| GEN-005 | 列表查询必须支持分页 | 🟢 低 |

---

### Step 6: 问题汇总与分级 (Issue Aggregation)
**目标**: 汇总所有发现的问题并分级

**输出**: `audit-report-[PRD名]-v[X].md`

**报告结构**:
```markdown
# PRD 审计报告

## 1. 审计概览
- PRD: [名称] v[版本]
- 审计日期: YYYY-MM-DD
- 审计人: AI Assistant
- 总体评级: [A/B/C/D]

## 2. 问题统计
| 级别 | 数量 | 状态 |
|------|------|------|
| 🔴 严重 | X | 待修复 |
| 🟡 警告 | X | 待修复 |
| 🟢 建议 | X | 可选 |

## 3. 严重问题 (Critical)
### C-001: [问题标题]
- **位置**: [章节/段落]
- **问题**: [描述]
- **影响**: [后果]
- **建议**: [修复方案]

## 4. 警告问题 (Warning)
...

## 5. 改进建议 (Suggestion)
...

## 6. 修复检查清单
- [ ] C-001 已修复
- [ ] C-002 已修复
- [ ] W-001 已修复
...
```

**评级标准**:
- **A (优秀)**: 无严重问题，≤2 警告
- **B (良好)**: 无严重问题，≤5 警告
- **C (需改进)**: 有严重问题或 >5 警告
- **D (不合格)**: 有多个严重问题

---

### Step 7: 修复验证 (Fix Verification)
**目标**: 验证 PRD 修复后的质量

**流程**:
1. 接收修复后的 PRD
2. 重新执行 Step 2-6（仅检查已修复项）
3. 更新审计报告状态
4. 如仍有严重问题，返回 Step 7

**通过标准**:
- 所有 🔴 严重问题已修复
- 修复后的评级 ≥ B

---

## ✅ 行动检查点 (Action Checklist)

### 物理级合规检查

- [ ] **文件命名**: `audit-report-[PRD名]-v[X].md`
- [ ] **位置正确**: 存入 `docs/audit/` 目录
- [ ] **关联记录**: 报告头部标注被审计 PRD 路径
- [ ] **索引更新**: 在 `docs/audit/INDEX.md` 登记
- [ ] **Git 提交**: message: "审计: [PRD名] v[X] - [评级]"

## 🧠 理性化表格 (Rationalization Table)

| 常见陷阱 | 对抗策略 | 检查方式 |
|---------|---------|---------|
| **遗漏上下文** | 强制读取项目背景和业务锚点 | 检查 audit-context.md |
| **规则库过时** | 定期更新 evolution-db.json | 检查规则日期 |
| **误报/漏报** | 使用多维度交叉验证 | 抽样人工复核 |
| **修复不彻底** | 强制闭环验证，不通过不结束 | Step 7 必须执行 |
| **评级主观** | 使用量化评分标准 | 对照评级标准表 |

## 📋 使用示例

### 示例: 审计登录功能 PRD

**输入**: `PRD-login-v1.md`

**审计发现**:

```markdown
## 严重问题
### C-001: 缺少验证码重发机制
- **位置**: 4.1 登录流程
- **问题**: 未定义验证码过期后如何重发
- **影响**: 用户可能无法完成登录
- **建议**: 添加"60秒后可重发"逻辑

### C-002: 未定义并发登录处理
- **位置**: 3. 用户故事
- **问题**: 同一账号多设备登录未说明
- **影响**: 安全风险
- **建议**: 明确是否允许多设备同时在线

## 警告问题
### W-001: 性能指标缺失
- **位置**: 5. 非功能需求
- **问题**: 未定义登录接口响应时间要求
- **建议**: 添加"登录接口 ≤ 500ms"
```

**输出评级**: C (需改进)
