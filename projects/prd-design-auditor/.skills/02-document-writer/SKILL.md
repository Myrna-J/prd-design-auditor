# 02-document-writer - 文档撰写师

> **版本**: 1.0.0
> **技能类型**: Core Skill
> **所属项目**: 产品经理AI助理

---

## 🎯 功能定位

将结构化的需求转化为标准化的 PRD 文档，确保格式统一、内容完整、易于理解。

---

## 🩺 触发条件 (Symptoms & Triggers)

当以下场景出现时，激活此技能：

- [ ] 需求分析师已完成结构化需求输出
- [ ] 需要生成正式 PRD 文档供评审
- [ ] 现有 PRD 格式混乱需要重构
- [ ] 多份需求需要合并为统一文档
- [ ] 需要输出不同格式的文档（Markdown/Word/PDF）

---

## 🛠️ 7步工作流 (7-Step Workflow)

### Step 1: 输入审查 (Input Review)
**目标**: 检查上游输出的完整性和质量

**输入**: `01-requirement-analyst` 的输出文件
**检查清单**:
- [ ] `raw-requirements.md` 存在
- [ ] `user-stories.md` 存在且用户故事 ≥ 1 条
- [ ] `feature-breakdown.md` 存在
- [ ] `user-flows.md` 存在（可选但推荐）
- [ ] `data-model.md` 存在（可选但推荐）

**质量门槛**:
- 用户故事必须包含验收标准
- 功能点必须有明确的输入/输出定义
- 优先级必须使用 MoSCoW 标注

---

### Step 2: 模板选择 (Template Selection)
**目标**: 根据项目类型选择合适的 PRD 模板

**可用模板**:

| 模板类型 | 适用场景 | 文件 |
|---------|---------|------|
| **标准PRD** | 常规功能开发 | `templates/prd-standard.md` |
| **技术PRD** | 架构/技术方案 | `templates/prd-technical.md` |
| **数据PRD** | 报表/数据分析 | `templates/prd-data.md` |
| **集成PRD** | 第三方对接 | `templates/prd-integration.md` |

**模板结构**（标准PRD）:
```markdown
# PRD: [产品/功能名称]

## 1. 文档信息
- 版本: v1.0
- 日期: YYYY-MM-DD
- 作者: [姓名]
- 评审人: [姓名]

## 2. 背景与目标
### 2.1 问题陈述
### 2.2 成功指标

## 3. 用户故事
### 3.1 主要用户
### 3.2 用户故事列表

## 4. 功能规格
### 4.1 功能清单
### 4.2 详细规格

## 5. 非功能需求
### 5.1 性能
### 5.2 安全
### 5.3 兼容性

## 6. 数据模型

## 7. 用户流程

## 8. 界面原型

## 9. 发布计划

## 10. 附录
```

---

### Step 3: 内容映射 (Content Mapping)
**目标**: 将结构化需求映射到 PRD 模板

**映射表**:

| PRD 章节 | 来源文件 | 映射规则 |
|---------|---------|---------|
| 2. 背景与目标 | `raw-requirements.md` | 提取背景、期望结果 |
| 3. 用户故事 | `user-stories.md` | 完整复制，保持格式 |
| 4. 功能规格 | `feature-breakdown.md` | 重组为表格+详情 |
| 5. 非功能需求 | `feature-breakdown.md` | 提取非功能部分 |
| 6. 数据模型 | `data-model.md` | 完整复制，添加ER图 |
| 7. 用户流程 | `user-flows.md` | 完整复制 |
| 8. 界面原型 | 新建 | 添加占位符或示意图 |

---

### Step 4: 文档撰写 (Writing)
**目标**: 生成完整的 PRD 文档

**输出**: `docs/prd/PRD-[功能名]-v1.md`

**撰写规范**:

#### 4.1 语言风格
- 使用**简洁中文**，避免冗长
- 技术术语保留英文（API, JSON, UUID）
- 使用主动语态："系统应该..." 而非 "应该被系统..."

#### 4.2 格式规范
- 标题层级：# ## ### ####（不超过4级）
- 列表：使用 - 或 1. 2. 3.
- 表格：用于对比、参数、状态
- 代码块：用于 API 定义、示例数据
- Mermaid：用于流程图、时序图

#### 4.3 内容深度
- **Must**: 必须实现的功能，详细描述
- **Should**: 应该实现的功能，中等详细
- **Could**: 可以实现的功能，简要描述
- **Won't**: 明确不实现的功能，说明原因

---

### Step 5: 交叉引用 (Cross-Referencing)
**目标**: 建立文档内部和外部链接

**引用类型**:

```markdown
## 内部引用
- 参见 [用户故事 US-001](#us-001)
- 详见 [数据模型](#6-数据模型)

## 外部引用
- 关联 PRD: [PRD-用户系统](./PRD-user-system-v2.md)
- 设计稿: [Figma - 登录页](https://figma.com/...)
- 技术方案: [RFC-001](../tech/rfc-001.md)
```

---

### Step 6: 格式审查 (Formatting Review)
**目标**: 确保文档格式统一、无错误

**检查清单**:
- [ ] 所有标题层级正确（不能跳级）
- [ ] 所有表格有表头
- [ ] 所有代码块有语言标记
- [ ] 所有链接可点击
- [ ] 文档有目录（TOC）
- [ ] 页眉页脚信息完整

**自动化检查**:
```bash
# Markdown 格式检查
markdownlint docs/prd/PRD-*.md

# 链接检查
markdown-link-check docs/prd/PRD-*.md
```

---

### Step 7: 输出交付 (Delivery)
**目标**: 生成多格式输出并归档

**输出文件**:

| 格式 | 文件名 | 用途 |
|------|--------|------|
| Markdown | `PRD-[功能名]-v1.md` | 开发参考 |
| Word | `PRD-[功能名]-v1.docx` | 商务评审 |
| PDF | `PRD-[功能名]-v1.pdf` | 正式归档 |
| HTML | `PRD-[功能名]-v1.html` | 在线浏览 |

**转换命令**:
```bash
# Markdown → Word
pandoc PRD-xxx.md -o PRD-xxx.docx --reference-doc=templates/word-template.docx

# Markdown → PDF
pandoc PRD-xxx.md -o PRD-xxx.pdf --pdf-engine=xelatex

# Markdown → HTML
pandoc PRD-xxx.md -o PRD-xxx.html --template=templates/html-template.html
```

## ✅ 行动检查点 (Action Checklist)

### 物理级合规检查

- [ ] **文件命名**: `PRD-[功能名]-v[版本].md`
- [ ] **位置正确**: 存入 `docs/prd/` 目录
- [ ] **索引更新**: 在 `docs/prd/INDEX.md` 登记
- [ ] **版本控制**: 同一功能多版本保留历史
- [ ] **关联记录**: 在 PRD 头部标注上游需求文档链接
- [ ] **Git 提交**: message: "PRD: [功能名] v[版本]"

## 🧠 理性化表格 (Rationalization Table)

| 常见陷阱 | 对抗策略 | 检查方式 |
|---------|---------|---------|
| **复制粘贴错误** | 使用脚本自动映射，减少人工复制 | 抽样检查关键数据 |
| **格式不统一** | 强制使用模板，自动化格式检查 | markdownlint 通过 |
| **遗漏更新** | 建立变更日志，每次修改记录 | 检查 CHANGELOG |
| **版本混乱** | 严格版本号规则，保留历史 | 文件命名检查 |
| **可读性差** | 强制图表、限制段落长度 | 人工阅读测试 |

## 📋 使用示例

### 示例: 将用户故事转化为 PRD

**输入** (`user-stories.md`):
```markdown
## US-001: 用户登录
**作为** 注册用户
**我希望** 使用手机号登录
**以便** 快速访问我的账户

**验收标准**:
- [ ] AC1: 输入正确手机号+验证码，登录成功
- [ ] AC2: 验证码错误，显示"验证码错误"
- [ ] AC3: 手机号未注册，提示"请先注册"
```

**输出** (`PRD-login-v1.md` 节选):
```markdown
## 3. 用户故事

### US-001: 用户登录
| 属性 | 内容 |
|------|------|
| 角色 | 注册用户 |
| 功能 | 使用手机号登录 |
| 价值 | 快速访问我的账户 |
| 优先级 | Must |

#### 验收标准
| 场景 | 操作 | 期望结果 |
|------|------|---------|
| 正常登录 | 输入正确手机号+验证码 | 登录成功，跳转首页 |
| 验证码错误 | 输入错误验证码 | 显示"验证码错误" |
| 未注册用户 | 输入未注册手机号 | 提示"请先注册" |

## 4. 功能规格

### 4.1 登录流程
```mermaid
flowchart TD
    A[输入手机号] --> B{格式正确?}
    B -->|否| C[提示"手机号格式错误"]
    B -->|是| D[发送验证码]
    D --> E[输入验证码]
    E --> F{验证通过?}
    F -->|否| G[提示"验证码错误"]
    F -->|是| H[登录成功]
```
```
