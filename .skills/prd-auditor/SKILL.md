## 🎯 技能描述

AI 驱动的产品需求文档（PRD）和 UI/UX 设计规范审计助手，用于检查文档的一致性、可行性和逻辑漏洞。

**v2.0 核心改进**:
- ✅ 细化审计维度至 7 大类 50+ 检查点
- ✅ 增加开发可行性评估
- ✅ 集成技术风险评估
- ✅ 提供可落地的改进建议模板
- ✅ 支持多文档关联审计

## 🛠️ 执行流程 (SOP)

### 阶段一：文档预检 (5 分钟)

```
1. 确认文档类型 (PRD / 设计规范 / 混合)
2. 检查文档基本结构完整性
3. 识别关联文档 (如有)
4. 确定审计深度和优先级
```

### 阶段二：深度审计 (核心)

```
1. 逐章节应用审计检查清单
2. 标记问题点并分类 (严重/中/低)
3. 交叉验证文档内部一致性
4. 识别逻辑漏洞和边界场景
5. 评估技术可行性和风险
```

### 阶段三：报告生成

```
1. 汇总问题清单
2. 计算各维度评分
3. 生成优先级排序
4. 提供改进建议和模板
5. 输出可跟踪的问题列表
```

### 阶段四：跟进闭环

```
1. 记录已修复问题
2. 复审修改后的文档
3. 更新审计状态
4. 归档审计报告
```

## 📐 审计检查清单 (v2.0)

### 一、PRD 完整性审计 (12 项)

| # | 检查项 | 检查要点 | 通过标准 |
|---|--------|----------|----------|
| 1.1 | 产品愿景 | 是否有清晰的产品目标和价值主张 | 一句话能说清产品解决什么问题 |
| 1.2 | 用户画像 | 目标用户群体是否明确 | 有具体的用户角色和场景描述 |
| 1.3 | 用户故事 | 是否用用户故事描述需求 | 符合"作为 X，我希望 Y，以便 Z"格式 |
| 1.4 | 功能列表 | 功能模块是否完整列出 | 有功能清单和优先级标注 |
| 1.5 | 业务流程 | 核心业务流程是否有流程图 | 有完整的流程图和文字说明 |
| 1.6 | 异常流程 | 异常场景是否覆盖 | 至少覆盖 80% 的异常分支 |
| 1.7 | 数据需求 | 数据字段和来源是否定义 | 有数据字典或字段说明 |
| 1.8 | 接口需求 | 内外接口是否明确 | 有关键接口定义 |
| 1.9 | 非功能需求 | 性能、安全等是否定义 | 有量化指标 |
| 1.10 | 依赖关系 | 外部依赖是否识别 | 列出所有外部系统和依赖 |
| 1.11 | 里程碑 | 关键节点是否明确 | 有时间线和交付物定义 |
| 1.12 | 验收标准 | 是否有可测试的验收条件 | 每个功能有明确的验收标准 |

### 二、业务逻辑审计 (10 项)

| # | 检查项 | 检查要点 | 通过标准 |
|---|--------|----------|----------|
| 2.1 | 流程闭环 | 业务流程是否有始有终 | 每个流程有明确的起点和终点 |
| 2.2 | 状态转换 | 对象状态变化是否清晰 | 有状态机或状态转换说明 |
| 2.3 | 条件分支 | 所有条件分支是否覆盖 | if-else 所有分支都有定义 |
| 2.4 | 边界条件 | 边界值是否定义 | 最大/最小/空值都有处理 |
| 2.5 | 并发处理 | 并发场景是否考虑 | 有并发冲突处理方案 |
| 2.6 | 事务一致性 | 数据一致性如何保证 | 有事务或补偿机制说明 |
| 2.7 | 超时处理 | 超时场景是否定义 | 有超时时间和处理逻辑 |
| 2.8 | 重试机制 | 失败重试是否定义 | 有重试次数和策略 |
| 2.9 | 降级方案 | 服务降级是否有预案 | 有关键功能的降级方案 |
| 2.10 | 回滚机制 | 操作回滚是否支持 | 有关键操作的回滚流程 |

### 三、技术可行性审计 (8 项)

| # | 检查项 | 检查要点 | 风险信号 |
|---|--------|----------|----------|
| 3.1 | 技术栈匹配 | 现有技术栈是否支持 | 需要新技术但未评估 |
| 3.2 | 性能指标 | 性能要求是否可实现 | QPS/响应时间超出常规 |
| 3.3 | 数据量评估 | 数据规模是否合理 | 未评估数据增长趋势 |
| 3.4 | 第三方依赖 | 第三方服务是否可靠 | 关键功能依赖单一服务 |
| 3.5 | 集成复杂度 | 系统集成难度评估 | 多系统对接但未定义协议 |
| 3.6 | 技术债务 | 是否有妥协方案 | 明确标注临时方案 |
| 3.7 | 扩展性 | 未来扩展是否考虑 | 预留扩展点 |
| 3.8 | 兼容性 | 历史数据/接口兼容 | 有兼容方案或迁移计划 |

### 四、权限与安全审计 (8 项)

| # | 检查项 | 检查要点 | 必须项 |
|---|--------|----------|--------|
| 4.1 | 角色定义 | 用户角色是否清晰 | ✅ |
| 4.2 | 权限矩阵 | 功能权限是否定义 | ✅ |
| 4.3 | 数据权限 | 数据访问权限是否定义 | ✅ |
| 4.4 | 认证方式 | 登录认证是否明确 | ✅ |
| 4.5 | 敏感操作 | 关键操作是否有二次确认 | ✅ |
| 4.6 | 审计日志 | 操作日志是否记录 | ✅ |
| 4.7 | 数据脱敏 | 敏感信息是否脱敏 | ✅ |
| 4.8 | 安全传输 | 数据传输是否加密 | ✅ |

### 五、数据与接口审计 (8 项)

| # | 检查项 | 检查要点 | 输出物 |
|---|--------|----------|--------|
| 5.1 | 字段定义 | 字段类型、长度是否定义 | 数据字典 |
| 5.2 | 必填校验 | 必填字段是否标注 | 校验规则 |
| 5.3 | 格式校验 | 数据格式是否定义 | 正则/格式说明 |
| 5.4 | 枚举值 | 枚举字段是否列出所有值 | 枚举清单 |
| 5.5 | 关联关系 | 表间关联是否定义 | ER 图/说明 |
| 5.6 | 接口协议 | 接口协议是否明确 | REST/RPC 等 |
| 5.7 | 请求响应 | 请求参数和响应格式 | 接口文档 |
| 5.8 | 错误码 | 错误码是否定义 | 错误码表 |

### 六、验收标准审计 (6 项)

| # | 检查项 | SMART 原则检查 |
|---|--------|---------------|
| 6.1 | 具体性 | 验收标准是否具体明确 |
| 6.2 | 可衡量 | 是否有量化指标 |
| 6.3 | 可实现 | 是否在能力范围内 |
| 6.4 | 相关性 | 是否与业务目标相关 |
| 6.5 | 时限性 | 是否有时间要求 |
| 6.6 | 可测试 | 测试用例是否可编写 |

### 七、设计规范审计 (8 项)

| # | 检查项 | 检查要点 |
|---|--------|----------|
| 7.1 | 设计语言 | 是否有统一的设计语言规范 |
| 7.2 | 组件复用 | UI 组件是否统一和复用 |
| 7.3 | 交互一致 | 相同操作是否有相同反馈 |
| 7.4 | 响应式 | 多端适配是否考虑 |
| 7.5 | 可访问性 | 是否符合无障碍标准 |
| 7.6 | 加载状态 | 加载/空/错误状态是否定义 |
| 7.7 | 反馈机制 | 操作反馈是否清晰 |
| 7.8 | 导航结构 | 信息架构是否清晰 |

## 📊 评分体系

### 综合评分计算公式

```
总分 = (完整性×20% + 逻辑×20% + 技术×15% + 安全×20% + 数据×15% + 验收×10%)

等级划分:
- A (90-100): 优秀，可直接进入开发
- B (75-89):  良好，需少量补充
- C (60-74):  合格，需中等修改
- D (40-59):  不合格，需大幅修改
- E (0-39):   不可用，需重新编写
```

### 风险等级定义

| 等级 | 标识 | 定义 | 响应要求 |
|------|------|------|----------|
| **严重** | 🔴 | 阻碍开发或存在重大风险 | 必须修复，否则不能进入开发 |
| **高** | 🟠 | 影响核心功能或用户体验 | 优先修复，建议进入开发前完成 |
| **中** | 🟡 | 影响部分功能或有潜在风险 | 开发过程中同步修复 |
| **低** | 🟢 | 优化建议或轻微问题 | 有时间再修复 |
| **提示** | 🔵 | 改进建议 | 后续迭代考虑 |

## 🚀 使用指南

### 触发方式

```
/prd-audit [文档路径]     # 审计单个文档
/prd-audit-full [项目路径] # 完整项目审计
/prd-audit-compare [PRD 路径] [设计路径] # 对比审计

### 输出选项

| 参数 | 说明 |
|------|------|
| `--detail` | 输出详细报告 |
| `--summary` | 仅输出摘要 |
| `--format=markdown|json|html` | 输出格式 |
| `--output=文件路径` | 输出到文件 |

### 审计深度配置

| 模式 | 适用场景 | 检查项 |
|------|----------|--------|
| **快速检查** | 日常迭代 | 核心 20 项 |
| **标准审计** | 版本发布 | 全部 50+ 项 |
| **深度审计** | 重大项目 | 全部 + 专家评审 |
