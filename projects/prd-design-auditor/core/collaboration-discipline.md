# 人机协作纪律与权限边界 (Collaboration Discipline & Boundaries)

> 本文件详细定义 AI 与指挥官在实际开发中"先做再议"和"环境保洁"的执行尺度，明确哪些行为由 AI 闭环处理，哪些行为必须等待指令。

## 🟢 AI 完全自治与闭环执行（放权区）
> **执行原则**: 直接执行、闭环反馈（不再反问“是否可以开始”或“你看可以吗”），完成后通过 `git commit` 等形式提交验收。

1. **常规代码与功能实现**: 在特定目录下（如 `src/`、`tests/`、特定项目模块）独立进行业务代码的增删改查、修复一般性逻辑 Bug。
2. **非破坏性的项目结构重构**: 根据“三维切割法则”进行规则文件迁移、模板（`templates/`）资产代码的克隆同步等操作。
3. **规范约定的文档更新**: 自主读取、写入、归档工作日志 (`work-log.md`) 和状态锚点 (`context.md`)，复盘时主动生成 `session-log-*.md` 等相关记忆资产更新。
4. **日常工具操作与 Git 流转**: 执行本地查询、搜索命令；自主撰写规范的 `commit message` 并自动完成 `git add` 和 `git commit`。
5. **明确无异议的业务指令**: 对指挥官提出的明确需求（如“把这个页面的底色改成暗改黑”），且在安全范围内，直接修改并完成快照留痕。

## 🔴 必须请求确认与人类兜底（红线区）
> **执行原则**: 必须暂停当前流程，主动抛出问题、等候统筹决断，或为指挥官生成好完整命令，**待达成共识或由人类亲自粘贴执行**。

1. **环境变更与底层依赖修改**（对应"环境保洁"铁律）:
   - **禁止**任何全局安装库的行为（如执行 `pip install xxx` 或 `npm install -g` 等）。
   - **禁止**更改底层操作系统环境变量。
   - **禁止**跨越当前项目工作区，修改操作系统核心及其他私密目录文件。
2. **物理打包与发版执行**:
   - 编译系统底层的二进制库应用。
   - 对生产环境的物理可执行文件进行构建、打包并推送到远端服务器或生产环境。
3. **强破坏性操作与核心宪法修改**:
   - 包含严重数据清空行为或大规模摧毁现有的核心业务模块。
   - 对 `global_rules.md` 或 `core/*.md` 这一级别“最高宪法协议”进行颠覆性的重组。
4. **存在验证型异议的情况**:
   - 如果 AI 研判指挥官的某一指令会导致严重 Bug、死循环或与上下文矛盾，AI 必须打断并提交“反对理由及改进建议”，**待明确达成共识后，方可执行该部分**。
